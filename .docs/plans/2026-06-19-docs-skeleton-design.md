# Docusaurus Documentation Skeleton — Design Plan

**Date:** 2026-06-19
**Status:** Approved for implementation

## Goal

Transform the internal `.docs/` content (written for LLM consumption) into a polished, welcoming Docusaurus site for end-user developers.

## Audience

Frontend/platform engineers embedding cross-domain iframes. Range: junior to senior. Tone: welcoming & approachable.

## Structure: 13 pages in 4 groups

### Introduction (onboarding)

1. **Home** (`index.md`) — Hero, value prop, features, comparison, code teaser
2. **Getting Started** (`01-getting-started.md`) — Install, first bridge, quick wins
3. **Core Concepts** (`02-core-concepts.md`) — Mental model, lifecycle, handshake, patterns

### Guides (learn by doing)

4. **Configuration** (`03-configuration.md`) — Full config reference grouped by category
5. **Type-Safe Bridge** (`04-typed-bridge.md`) — `createTypedIframeBridge`, contract maps
6. **Wire Protocol** (`05-wire-protocol.md`) — Envelope spec, iframe integration guide
7. **Security** (`06-security.md`) — Security model, profiles, CSP, sandbox, checklist
8. **Use Cases & Recipes** (`07-use-cases.md`) — Copy-pasteable config recipes
9. **Debugging & Diagnostics** (`08-debugging.md`) — Diagnostic recorder, logger hooks, debug mode

### Reference (look up)

10. **API Reference** (`09-api-reference.md`) — Hand-written reference for all public exports
11. **Error Codes** (`10-error-codes.md`) — Complete error code table with recovery actions

### Help

12. **Troubleshooting** (`11-troubleshooting.md`) — Diagnostic flowcharts and checklists
13. **FAQ** (`12-faq.md`) — Common questions

## Sidebar

Explicit sidebar in `sidebars.js` — no autogenerated mode. Four categories matching the groups above.

## Implementation Approach

1. **This session:** Create full skeleton (all `.md` files with frontmatter `todo` lists and placeholder sections)
2. **Future sessions:** Refine pages one-by-one, checking off `todo` items in frontmatter
3. Each refinement session verifies `npm run docs:build` passes

## Key Principles

- Example-first, progressive depth
- Every concept: what → why → how
- Code snippets tested against actual SDK API
- Mermaid diagrams for lifecycle and handshake
- Docusaurus admonitions (tip/warning/danger) for security guidance
- Hand-written API reference (no TypeDoc auto-generation)
